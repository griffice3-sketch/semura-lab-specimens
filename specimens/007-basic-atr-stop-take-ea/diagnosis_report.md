## 【前提】本ロジックの目的・狙い・意図・思想の解剖

コード全量から逆算される設計思想：

- **コア機能**：ATRを用いた動的SL/TP計算の実装デモンストレーション。エントリーシグナルの雛形は存在せず、「ポジションがゼロなら即時発注」という**無条件エントリー**が実装されている。
- **方向性**：初期化時に`Ordertype`（_buy / _sell）を手動選択。以降は**単一方向のみ**を繰り返す。
- **SL/TP設計**：`ATR × SLfactor（1.5）` / `ATR × TPfactor（2.0）`。RR≒1:1.33と、ATRの動的スケーリングで相場環境にある程度適応する思想。
- **設計意図の正体**：ATRベースのSL/TP機構を実装したい開発者向けの**骨格コード（スケルトン）**。心臓（エントリーロジック）は意図的に空洞。

**本検体は戦略ではなく部品集合体である。**

## 1. 総合所見（生存能力の有無とその理由）

**判定：現状では生存能力ゼロ。ただし骨格の一部は再利用に値する。**

理由を三点。

**① 無条件エントリー＝ランダムエントリーと等価**
`if(Position()==0) { SendOrder(); }` ―― 現在ポジションが無ければ毎ティックで発注する。SLまたはTPで決済されると次のティックで即座に再エントリー。**これは方向付きのランダムウォーク**であり、エッジは存在しない。固定方向（Buy only / Sell only）のため、トレンド転換時には延々と逆方向へ打ち続ける。

**② RRは1:1.33と見かけ上良好だが意味がない**
TPfactor=2.0 / SLfactor=1.5でRR≒1:1.33。しかしエッジのないエントリーにRRを掛けても期待値はスプレッド分だけマイナスになる。「良いRRに見えるが中身がない」典型。

**③ 骨格としての資産価値は存在する**
ATRベースSL/TP・単一ポジション管理・ロットサイズ正規化・証拠金チェック・マジックナンバー管理は機能する実装だ。これらは適切なエントリーロジックと組み合わせれば再利用可能。

## 3. 致命的脆弱性・CAP（過剰最適化／誤魔化し）の疑い

### コードレベルの致命傷

**▼ V-1：エントリー条件の完全不在（最重要）**
作者が自ら認めている通り、エントリーロジックが存在しない。これは「脆弱性」ではなく**設計上の空洞**。外科的修正不能。新規実装が必須。

**▼ V-2：`Position()`関数の論理バグ（中程度・実運用で誤動作）**

```cpp
// 現状（バグあり）
if(OrderSymbol()!=Symbol() && OrderMagicNumber()!=MagicNumber)
    continue;

// 正しい実装
if(OrderSymbol()!=Symbol() || OrderMagicNumber()!=MagicNumber)
    continue;
```

`&&`（AND）を使用しているため、**「シンボルが違う かつ マジックが違う」場合のみスキップ**される。同一シンボル・異なるマジック番号の他EAポジションを自己ポジションと誤認識し、「ポジションあり」と判断してエントリーを止める誤動作が発生し得る。`||`（OR）に修正が必要。

**▼ V-3：SL/TPの二段階設定による無防備区間**
`OrderSend`では`SL=0, TP=0`で発注し、その後`OrderModify`でSL/TPを設定する二段構え。**OrderSend成功からOrderModify完了までの間、ポジションはSL/TP無しの状態で存在する**。この瞬間に急激な価格変動が起きた場合、無制限の損失にさらされる。発注と同時にSL/TPを指定するのが原則。

**▼ V-4：iATR(... , 0)の形成中バー参照**
`iATR(Symbol(), atrTf, atrPer, 0)` ―― 未確定の現在バーのATRを使用。SL/TP計算値がバー内でわずかに変動する。エントリーロジックのリペイントほど致命的ではないが、確定足（index 1）基準が望ましい。

**▼ V-5：固定単一方向の構造的リスク**
BuyまたはSellのどちらか一方に固定されたまま動作する設計は、相場環境の変化に完全に無防備。下降トレンドで`_buy`固定なら延々と損切りを繰り返す。双方向適応またはトレンドフィルタが必須。

**▼ V-6：スプレッド・時間帯フィルタの完全不在**
無条件エントリーのため、スプレッド拡大時間帯・指標発表前後・流動性希薄帯に関わらず発注する。スキャルピング規格でのコスト管理は全く考慮されていない。

### CAPの疑い

**現時点で適用不能（データ未提出）。**
ATRの2パラメータ（atrPer・factor類）は最適化対象になり得る。今後データが提出された際、これらが既定値（14, 2.0, 1.5）から大幅に逸脱していれば即CAPチェック対象。

## 4. 処方箋（Semura Lab規格への再生指示）

作者が意図した通り「発展のための基礎テンプレート」として扱い、使える部品と構築すべき部品を明示する。

### 【使える部品（流用可）】
1. **ATRベースSL/TP機構** ――ただし後述の修正を施した上で。動的SL/TPとしての思想は有効。
2. **マジックナンバー管理** ――Magic + 方向の分離は参考にできる構造。
3. **ロットサイズ正規化（LotSize関数）** ――`MODE_LOTSTEP`基準の正規化は実装として正しい。
4. **CheckMoneyForTrade** ――証拠金チェックは適切。

### 【構築が必要な部品】

**① エントリーロジックの新規実装（最優先）**
無条件発注を廃棄し、技術的優位性に基づくシグナル条件を設計。Semura Lab規格であれば、スキャルピング向けの高精度・短期モメンタム系または値動き統計系を推奨。

**② `Position()`のバグ修正**
```cpp
// 修正版
if(OrderSymbol()!=Symbol() || OrderMagicNumber()!=MagicNumber)
    continue;
```

**③ SL/TP発注の一体化**
```cpp
// OrderSend時にSL/TPを直接指定
Ticket = OrderSend(Symbol(), _mode, LotSize(), pricemode, 5,
                   NormalizeDouble(_SL, Digits),
                   NormalizeDouble(_TP, Digits),
                   WindowExpertName(), MagicNumber, 0, col);
// OrderModifyは不要になる
```

**④ ATR参照を確定足へ変更**
```cpp
atr = iATR(Symbol(), atrTf, atrPer, 1);  // 0→1へ
```

**⑤ スプレッド・時間帯フィルタの実装**
```cpp
double spread = (Ask - Bid) / _point;
if(spread > 0.6) return;  // 摩擦超過時はエントリー回避
// + 時間帯フィルタ（スプレッド拡大時間帯を外部パラメータで指定）
```

**⑥ SL/TPをSemura Lab規格（5〜8pips RR1:1）へ設計**
ATRベースの動的幅はSemura Lab規格の極小固定レンジと相性が悪い局面がある。ATRをフィルタ（ボラ環境判断）として使い、SL/TP自体は固定pips制に変更することを検討せよ。

**⑦ 双方向対応の追加**
固定方向を廃棄し、エントリーシグナルに応じてBuy/Sellを動的に選択する構造へ。
