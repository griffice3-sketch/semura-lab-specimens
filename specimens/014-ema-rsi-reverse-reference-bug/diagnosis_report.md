## 0. 前提解析｜ロジックの目的・狙い・思想

解剖の結果、本検体の設計思想は以下と判定する。

- **狙い:** 短期EMA(12)と長期EMA(26)のゴールデン／デッドクロスを「トレンド発生の起点」と捉え、RSI(14)によるモメンタム順張りフィルター（買い≧55／売り≦45）で「ダマシ」を間引く古典的トレンドフォロー。
- **思想:** 損小利大（SL30 / TP60、RR=1:2）を骨格とし、ブレイクイーブン（15pips）とトレーリング（20pips）で利益確定の機動性を持たせる。
- **資金管理思想:** 固定ロットではなく **「Equity×RiskPercent / SL」によるリスク基準サイジング** を採用。単一ポジション縛り（`!hasPos`）でナンピン・マーチンの排除を明示。

**所見:** 思想そのものは「自然の摂理」に概ね合致した健全な設計哲学である。ナンピン排除・リスク基準ロット・確定足参照という骨格は評価に値する。**問題は思想ではなく実装と検証の欠落にある。**

## 1. 総合所見（生存能力の有無）

**生存能力：現状ゼロ。ただし骨格は救済可能。**

本検体は「思想は健全、実装は致命傷、検証データは皆無」という三重苦の状態にある。

最大の問題は**第2軸（論理の誠実性）以前に、シグナル抽出ロジックが時間軸を逆参照している重大バグ**を抱えている点だ（詳細は §3-A）。これにより、本検体は設計者が意図した「ゴールデンクロスで買う」ではなく、**実質的に逆方向のシグナルでエントリーしている可能性が極めて高い**。

加えて、**バックテストデータ・PF・月利・勝率・WFA結果が一切添付されていない。** 数値が存在しない以上、月利10%もPF1.2〜1.5も「自己申告すらない空欄」である。診断医として、空欄に対してPASSを出すことはできない。

## 3. 発見された致命的脆弱性

### A.【最重要・致命傷】CopyBuffer の時系列逆参照バグ

```cpp
double fastBuf[2], slowBuf[2];
CopyBuffer(hFastMA, 0, 1, 2, fastBuf); // ArraySetAsSeries 未設定
double fastNow  = fastBuf[0];
double fastPrev = fastBuf[1];
```

`ArraySetAsSeries()` を呼んでいないため、受信配列は**昇順（index0=より過去、index1=より現在）**で格納される。
すなわち実際には：

- `fastBuf[0]` = 2本前の足（より過去）
- `fastBuf[1]` = 1本前の確定足（より現在）

**コードはこれを `fastNow=fastBuf[0]`／`fastPrev=fastBuf[1]` と取り違えている。** 結果、`bullCross` 判定の時間順序が反転し、**設計者が「上抜けクロス」と信じている条件が、実体としては「下抜けクロス」を検知している。** RSIフィルター（買い≧55）と組み合わさることで、シグナルが自己矛盾を起こし、エントリーは激減または完全な逆張り誤動作となる。

**これは過剰最適化以前の「製品としての不良」である。** バックテスト数値が仮に存在しても、この実装である限りその数値は無意味。

### B. スプレッド摩擦・拡大時間帯の無防備

- `MaxSpread_Points=30`（=3.0pips）は摩擦上限として緩すぎ、Semura規格の0.6pips摩擦想定と乖離。
- ロールオーバー帯・指標前後のスプレッド爆発を**時間帯で遮断する設計が皆無**。`TradingWindow()` は単なる時刻フィルターであり、摩擦回避思想を内包していない。
- SL30/TP60に対し0.6pipsの往復摩擦を載せると、RR・期待値は確実に劣化するが、その検証が行われた形跡なし。

### C. ポジション選択にマジックナンバー・フィルターなし

`PositionSelect(_Symbol)` は同一シンボルの**全ポジション**を掴む。同一口座・同一通貨で他EAや手動ポジションが存在する場合、本EAが他人の建玉を改変・誤管理する。リアル口座運用での事故要因。

### D. リスクロットとSL距離の不整合

`LotSizeByRisk(SL_Pips)` は固定30pipsで計算するが、発注時に `minStop`（StopsLevel）でSLが拡張される分岐が存在する。SLが広がってもロットは30pips基準のまま → **想定リスクをわずかに超過する**。微小だが「リスク管理の誠実性」の観点で減点。

## 4. 処方箋｜Semura Lab規格への再生指示

優先度順に施術せよ。

**【最優先・必須】**
1. **逆参照バグの根治。** 全indicator配列に `ArraySetAsSeries(buf, true)` を適用し、`index0=最新確定足` に正規化せよ。修正後、`fastNow=buf[0]`／`fastPrev=buf[1]` が正しい時系列となる。**この修正なしに以降の議論は全て無効。**

2. **エビデンスの提出。** 13年間（または最低でも複数年）のティック品質バックテスト、WFA分割結果、PF・最大DD・月次リターン分布を添付せよ。**数値なき検体に採用判定は下さない。**

**【規格適合のための施術】**
3. **スプレッド設計の刷新。**
   - `MaxSpread_Points` を実運用ブローカーの平常値＋余裕（目安1.0〜1.5pips相当）まで引き締め。
   - ロールオーバー帯（サーバー時間23:55〜00:10等）・早朝薄商い帯を**エントリー除外時間帯**として明示実装。
   - バックテストは0.6pips摩擦（またはリアルティックスプレッド）を必ず織り込むこと。

4. **マジックフィルター追加。** `PositionGetInteger(POSITION_MAGIC)==MagicNumber` を全ポジション操作の前提条件に組み込み、他建玉との隔離を担保せよ。

5. **リスク整合性の修復。** 発注直前に確定した実SL距離を用いてロットを再計算し、想定リスク超過を排除せよ。

6. **期待値の現実検証。** 修正後、トレーリング・ブレイクイーブンの早期発動が「利大」を削っていないか、RR実効値とPFで検証すること。

## 5. 総合評価

# 判定：**差し戻し（再構築可）**

**論拠:**
- 設計思想（単一ポジション・リスク基準・確定足参照・損小利大）は **Semura規格と矛盾せず救済価値あり**。よって「不採用」とはしない。
- しかし §3-A の時系列逆参照バグは **検体が現状「設計意図と逆に動く欠陥品」** であることを意味し、このままの採用は論外。
- さらにバックテスト一切なし・スプレッド時間帯設計なしで、必須条件の半数が事実上FAIL／要検証。

**結論：** 骨格は生かせる。よって解体（不採用）ではなく **差し戻し**。ただし、§4-1（バグ根治）と§4-2（エビデンス提出）が満たされるまで、本検体は**再診の受付対象外**とする。逆参照バグを修正した「同一思想の別個体」を、数値を揃えて再提出せよ。

> 数字を出さない検体は、優位性が無いのではない。**そもそも生きているかどうかすら証明していない。**
