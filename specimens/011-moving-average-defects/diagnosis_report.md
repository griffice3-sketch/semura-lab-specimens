## 0. 検体の目的・狙い・思想の解析

コードから逆算される設計思想は以下の通り。

- **エントリーロジック:** SMA(14)に対し、前足(`[1]`)が「始値→終値」でMAを跨いだ瞬間（クロス）に順方向エントリー。終値がMAを上抜けで買い、下抜けで売り。
- **エグジット思想:** SL 100 / TP 800という**1:8の超非対称ペイオフ**。「損小利大、トレンドの大波を取りに行く」典型的なブレイクアウト追従思想。
- **建値移動(ブレイクイーブン):** 2段階の建値移動でトレイルを意図。
- **時間フィルタ:** 3時〜22時（サーバー時間）のみ稼働。

**所見:** 思想自体は「低勝率・高RRのトレンドフォロー」という古典的かつ自然の摂理に反しないコンセプト。だが、**思想と実装が完全に乖離している。** コードは作者の意図通りに1ミリも動いていない。これは戦略の良否以前の問題である。

## 1. 総合所見（生存能力の有無）

**生存能力: なし（実装段階で死亡）**

本検体はバックテスト結果以前に、**コードが論理的に破綻しており「作者の意図した動作」を一切実行していない**。優位性の検証に進む資格すら持たない。具体的には、

1. **pips/point換算の致命的誤り**により、SL/TPが意図の1/10で発注されている。
2. **建値移動ブロックが`OrderSelect()`を経ずに実行**され、未定義のオーダー情報を参照している（暴発・誤作動の温床）。
3. **`OrderModify()`の引数順序が崩壊**しており、SLとTPに異常値を流し込んでいる。
4. **単一ポジション保証ロジックが存在しない**（ポジション数チェックなし）。
5. **スプレッドフィルタが皆無**。

これは「調整可能なロジック」ではなく「動作しないコード」である。バックテストデータが添付されていれば、それは**バグの産物であり、市場の優位性を一切反映していない（＝CAP相当の虚偽数値）**。

## 3. 発見された致命的脆弱性

### 脆弱性①：pips → point の換算ミス【最重要】
```
StopLossPips * MarketInfo(Symbol(), MODE_POINT)
```
`MODE_POINT`は5桁業者で0.00001。`100 * 0.00001 = 0.001`＝**実質10pips**。
→ SL=10pips、TP=80pips で発注されている。作者が信じる「100/800」は完全な幻想。正しくは`MODE_POINT * 10`、または`SymbolInfoDouble(...,SYMBOL_TRADE_TICK_SIZE)`基準の明示的pip定義が必須。

### 脆弱性②：建値移動が「オーダー未選択」で実行
`CheckForOpen()`後半の建値移動ブロックは、**`OrderSelect()`を一度も呼ばずに** `OrderType()` / `OrderOpenPrice()` / `OrderTicket()` を参照している。これらは直前に選択された（あるいは無効な）オーダーを指し、**他シンボル・他EAのポジションを誤改変する危険すらある**。動作は完全に未定義。

### 脆弱性③：`OrderModify()`の引数順序崩壊
```
OrderModify(OrderTicket(), OrderOpenPrice(), breakevenPrice1, newStopLoss1, 0, clrWhite)
```
正規シグネチャは `OrderModify(ticket, price, stoploss, takeprofit, expiration, color)`。
本コードは**第3引数(stoploss)にbreakevenPrice1、第4引数(takeprofit)にnewStopLossを渡している**。つまりSLとTPを取り違え、TPに異常値を上書き。建値移動どころかポジションのTPを破壊する。

### 脆弱性④：単一ポジション保証の欠如
エントリー前に`OrdersTotal()`や自MagicNumberの保有数を数えていない。新バーでクロス条件が連続成立すれば多重建玉となる。「ナンピン禁止」は表面的に守れていても、リスク多重化という形で摂理に反する。

### 脆弱性⑤：スプレッド・スリッページ無防備
- スプレッド拡大時間（早朝・指標時）の回避なし。
- `OrderSend`のslippage=0、SL/TP発注値にスプレッド非考慮。
- `CheckForClose()`はブローカーSL/TPと**二重管理**で冗長かつ約定価格の整合性を崩す。

### 脆弱性⑥：資金管理の不在
固定0.5ロット。複利・残高比例なし。「月利10%」を語る土台が存在しない。

## 4. 処方箋（Semura Lab規格への再生指示）

**動作するまで戦略評価には進ませない。** 以下を順守せよ。

1. **pip定義の明示化**
   ```
   double pip = (_Digits==3 || _Digits==5) ? _Point*10 : _Point;
   ```
   全SL/TP/建値計算を`pip`基準へ統一せよ。

2. **建値移動ロジックの分離・再構築**
   建値移動は`CheckForOpen()`から完全に切り離し、独立関数化。必ず`OrderSelect()`でループ選択した上で、`OrderModify(ticket, OrderOpenPrice(), 新SL, OrderTakeProfit(), 0, clr)`と**引数順序を厳守**せよ。TPを破壊するな。

3. **単一ポジション・ガードの実装**
   エントリー前に自Magic・自Symbolの保有数をカウントし、`count==0`時のみ`OrderSend`を許可せよ。

4. **スプレッドフィルタの実装**
   ```
   double spread = (Ask - Bid)/pip;
   if(spread > 0.6) return; // 0.6pips摩擦を超えたらエントリー禁止
   ```
   加えて早朝ロールオーバー時間帯のエントリー除外を明記せよ。

5. **二重決済の排除**
   ブローカー側SL/TPを使うなら`CheckForClose()`の手動決済は削除。トレイル/建値のみを能動管理せよ。

6. **資金管理の導入**
   リスク%ベース（例：残高の1〜2%をSL距離で割ってロット算出）へ移行。固定ロットでは月利目標は議論に値しない。

7. **再提出時の必須添付**
   修正後、**スプレッド込み・リアルティック・13年WF**の結果を添えて再提出せよ。1:8ペイオフは「TP到達率」が生命線であり、現状の固定値には何の論理的根拠もない。最適化ではなく**仮説に基づく根拠**を示せ。
