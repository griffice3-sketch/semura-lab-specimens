## 1. 総合所見（生存能力の有無とその理由）

**判定：生存能力ゼロ。優位性の議論以前に、エッジが数学的に存在しない。**

理由を三点、順に示す。

**① エッジがゼロである数学的証明**
ランダムエントリー（各ティックで独立した乱数）は、長期的に勝率50%に収束する。本EA既定のTP=100/SL=200はリスクリワード**1:0.5**（損失方向のRR）。損益分岐点の必要勝率は：

```
必要勝率 = SL / (SL + TP) = 200 / (200 + 100) = 66.7%
```

ランダムエントリーの期待勝率50%に対し、66.7%が必要。**期待値は構造的にマイナス**。スプレッドコストを加算すれば更に悪化。これは戦略の良否ではなく、算数の問題だ。

**② 最大10ポジション同時保有の爆発リスク**
`MaxOpenTrade = 10`。買い・売りが異なるランダムシンボルで同時に最大10本積み上がる。リスクは予測不能に連鎖し、「リスク管理」として設置された5%ストッパーが機能する前に連鎖損失が走る。

**③ 検証不能なバックテスト**
ランダムシンボル・ランダムロット・ランダムエントリーの組み合わせは、同一パラメータで実行しても毎回異なる結果を生む。**バックテストの再現性が原理的に存在せず、WFAの概念が適用不能**。

## 3. 致命的脆弱性・CAP（過剰最適化／誤魔化し）の疑い

### ロジックレベルの致命傷

**▼ V-1：エントリー根拠の完全不在（最重要・修復不能）**
全エントリーが乱数。市場情報を一切参照しない。これは「脆弱性」ではなく**設計の根本的欠如**。改修ではなく全面再構築に相当する。

**▼ V-2：毎ティック`MathSrand(GetTickCount())`の再シード**
乱数生成器を毎ティック再初期化している。同一ミリ秒内の連続ティックでは同じシードから同じ乱数が生成され、連続して同一方向に発注し続けるリスクがある。乱数の質が意図せず偏る。

**▼ V-3：RR逆転構造（TP100/SL200）**
仮に技術的エッジがあったとしても、このRRでは勝率66.7%以上が必要。スキャルピング規格（RR1:1、5〜8pips）とは真逆の設計。

**▼ V-4：ランダムシンボルの危険性**
`randomsymbol()`がMarket Watchの全シンボルからランダム抽出。XAUUSDが選ばれれば1ティックの変動幅がFXの数十倍。ロットサイズもランダムであるため、**1トレードで口座を飛ばす組み合わせが確率的に発生し得る**。

**▼ V-5：SDG（スーダンポンド）の混入**
`randomsymbol_pairs[]` に `"SDG"`（スーダンポンド）が含まれる。これは主要FX市場で取引されない通貨。シンボルマッチングロジックにより、証券会社によっては意図しないシンボルへの発注が発生する。

**▼ V-6：5桁Brokerでの`Point`未正規化**
SL/TP計算に `* Point` を使用。5桁ブローカー（Point=0.00001）では `200 * Point = 0.002 = 20pips`、4桁ブローカー（Point=0.0001）では `200 * Point = 0.02 = 200pips`。**ブローカーの桁数によって実際のSL/TPが10倍異なる**。正規化処理が欠如。

**▼ V-7：リスク管理の脆弱性**
「含み損が残高の5%に達したら全決済」は一見まともだが、10本ポジションが同時に異なるシンボルで保有されている状況では、**急激な市場変動時に5%トリガーが追いつかず一気に10〜20%の損失が発生し得る**。単一ポジションを前提とした設計ではない。

### CAPの疑い

**性質上、CAPは適用不能。** ランダムエントリーにパラメータ最適化を施しても、それは相場構造への適合ではなく純粋な過去データへの偶然フィット。バックテスト結果を「最適化」で美化すること自体が意味を持たない。**もし良好なバックテスト結果が提出された場合、それは100%偶然の産物であり、OOSでの再現性はゼロに収束する。**
