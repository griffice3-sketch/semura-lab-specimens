## はじめに：このEAは何をするロジックか

本レポートは、海外のMQL4コミュニティで広く流通している自動売買EA（エキスパートアドバイザー）1本を、ソースコード全文（1,047行）から技術的に解剖した記録である。広告や成績画像ではなく、コードの実態だけを物証とする。

### 1. ロジックの仕様・狙い・思想（事実）

このEAが実装している取引ロジックの骨格は、コードから以下の通り読み取れる。

- **時間軸とトレンド判定:** M5（5分足）で動作。期間204のEMA（指数移動平均）の傾きで相場の大局（上昇/下降トレンド）を判定する。
- **エントリーシグナル:** Stochasticオシレーター（13,3,3）が買われすぎ（80超）・売られすぎ（20割れ）の領域でクロスした点を「過熱」として検知。その後、短期EMA（M1の4EMA）の加速方向がトレンドと一致したらエントリーする。
- **利確（TP）:** ボラティリティ（ATR）に応じて可変。通常時は5pips、低ボラティリティ時は2pipsという極小利確。
- **損切（SL）:** 固定で30pips。
- **補助フィルタ:** スプレッド上限・取引時間帯・最小エントリー間隔・ForexFactoryのRSSフィードを使ったニュース回避機能を備える。

**設計思想:** この仕様から読み取れる狙いは明白だ。「移動平均で大局を判断し、オシレーターで過熱感を測り、数pipsの小さな利益を高い確率でかすめ取る。逆行に対しては30pipsという広い含み損を許容して耐える」——典型的な**利小損大（利益は小さく・損失許容は大きく）型のトレンドフォロー・スキャルピング**である。利確2〜5pipsに対し損切30pips、すなわち**リスクリワードは1:6〜1:15**。

### 2. このロジックが「勝てない」と予測される理由

このEAは、バックテスト（過去検証）上では高い勝率と美しい右肩上がりの資産カーブを描きやすい設計である。だがその構造には、リアル口座（実弾運用）で機能しないと予測される理由が、コードの仕様レベルで複数存在する。以下は本編の物証・数理に入る前の、構造から導かれる予測である。

- **「コツコツドカン」を生む利小損大構造。** 利確2〜5pipsに対し損切30pips。1回の損切りを取り返すのに6〜15回の連勝が必要になる。「EAでコツコツ勝っていたのに、たった一度の負けで口座資金が大きく減った／飛んだ」という典型的な事象は、このリスクリワード構造から構造的に生じる。高い勝率は優位性の証明ではなく、損切りを遠くに置いた代償にすぎない。
- **極小利確とスプレッドの関係。** 利確目標が2〜5pipsという極小の世界では、スプレッド（例：0.6pips）やスリッページが利益に占める割合が大きい。利確2pipsならスプレッド0.6pipsだけで利幅の約30%が消える。「バックテストでは勝てるのに、実口座（リアル）ではジリ貧になる・成績が一致しない」という乖離の主因はここにあると予測される。
- **遅行指標の宿命。** Stochasticや移動平均（EMA）は過去の値から計算される遅行指標であり、シグナルが確定する頃には値動きの初動は終わっていることが多い。出遅れた位置から数pipsを抜きに行く行為は、エントリー直後の自然な押し戻し（プルバック）のノイズに刈られるリスクを高める。
- **「バックテストでは動くのに実口座で再現しない」要因。** ニュース回避に使うRSS取得（WebRequest）はMT4のストラテジーテスターでは動作しない。バックテストと実機で前提が変わる構造が、コードの仕様段階で存在する。

これらは「予測」である。果たしてコードの実態は、この予測通りなのか——あるいは予測を超えて壊れているのか。以下、全1,047行の監査結果を、物理的・数学的な事実のみで示す。

## 0. 総合所見（先に結論）

**この検体は「ラボ規格に矯正したら崩壊する」以前に、現状のコードのまま既に複数の致命的バグで機能不全に陥っている。** 「RR矯正で崩壊」「極小TPがスプレッドで蒸発」という想定は数理的に正しい（§3-4で証明）。だがそこに到達する前に、**そもそもこのEAは設計者の意図通りに動いていない**。

しかも重要な発見が一つ——**コードに付随する説明文と実コードが一致していない**。説明文は「price action」「206 EMA」を謳うが、コードに価格アクション判定は無く、EMAは204。**説明文がコードの実態を表していない（実装と乖離した仕様記述）。**

CAP（意図的なズル）の観点では：**露骨な未来リーク（peeking）は1箇所、構造的に存在する**（§5-1、`shift=0`の未確定足参照）。ただしこれは「バックテストを良く見せる悪意の細工」というより「初心者がやりがちな実装ミス由来のリーク」の色が濃い。プラセボ（無意味な指標の羅列）は複数該当。

## 1. 検体の構造概観（何をしようとしているか）

事実として、コードが実装している取引ロジックの骨格：

- **時間軸:** M5（`DEF_TIMEFRAME=PERIOD_M5`）。トレンド判定に EMA204(M5)、エントリー速度判定に EMA4(M1)。
- **トレンド方向:** `maPrev` vs `maCurr`（204EMAの傾き）でトレンド方向を判定。
- **シグナル:** Stochastic(13,3,3) が上限80超/下限20割れ + クロス（`checkCrossing`）で「マーキング」。
- **エントリートリガ:** マーキング後、EMA4(M1)の傾き速度 `fastMaSpeed` が閾値超で、タイムアウト30分以内なら発注。
- **TP:** ATRが閾値未満なら2pips、以上なら5pips（`GetNormalizedTakeProfit`）。
- **SL:** 固定30pips（`EA_SL_PIPS`）。
- **フィルタ:** スプレッド上限3、ATR下限、取引時間5-24時、最小間隔15分、ニュース回避45分。
- **ポジション:** Long/Short各1（`EA_MAX_OPEN_TRADES=1`）。

**設計思想の解釈:** 「204EMAでトレンド方向を決め、Stochの行き過ぎ＋クロスで過熱を検知し、短期EMAの加速でトレンド方向に乗って、小さく利確（2-5pips）・大きく耐える（SL30pips）」——典型的な**利小損大（マーチン的ではないが、損切りを遠くに置いて勝率を稼ぐ）**プロファイル。これがラボ規格(RR1:1)と正面衝突する核心。

## 2. 致命バグの解剖（規格を当てる前に既に壊れている）

### 2-1. 【FATAL-A】決済条件が数学的に永遠に成立しない

最も深刻なバグ。利確決済ロジックが**論理矛盾で絶対に発火しない**。

```cpp
// 504-509行（Long決済）
if((orderLong[i].IsOpen()) &&
   ((lastAtr < EA_ATR_VOLATILITY_TRH) && (orderLong[i].GetGrossProfitPips() >= EA_TP_PIPS_LV)) &&
   ((lastAtr > EA_ATR_VOLATILITY_TRH) && (orderLong[i].GetGrossProfitPips() >= EA_TP_PIPS)))
  {
   orderLong[i].CloseTrade();
  }
```

条件を抽出すると、決済には以下の**両方が同時に真**である必要がある：

- `(lastAtr < TRH) AND (利益 >= 2pips)` … 条件X
- `(lastAtr > TRH) AND (利益 >= 5pips)` … 条件Y

しかし `lastAtr < TRH` と `lastAtr > TRH` は**同時に真になり得ない**（同一の `lastAtr` が閾値より小さく、かつ大きいことは不可能）。

> **物理的帰結:** `X AND Y` は恒偽（常にfalse）。**この利確決済ブロックは、いかなる相場状況でも一度も実行されない。** 設計者は明らかに `OR`（低ボラなら2pips利確、高ボラなら5pips利確）を意図したが、`AND`で書いたため決済ロジックが完全に死んでいる。

**ではどうやって決済されるのか？** → `OpenLong/OpenShort` 内の `OrderModify`（250,282行）でサーバー側にTP/SLが設定されている。つまり**実際の決済はサーバー側のTP/SL約定に依存**し、この `CheckTrade` 内の能動的利確ロジックは完全な死にコード。

- 影響①: 「ATRに応じた動的利確」という設計意図は**実際には機能していない**。サーバーTP（`GetNormalizedTakeProfit`が設定した値）だけが効く。
- 影響②: ただし `GetNormalizedTakeProfit` 側（342行）は三項演算子で正しく `OR` 相当の分岐をしているため、**サーバーTPは正しく機能する**。結果として「動的利確の二重実装のうち、能動側が死に、受動側だけ生きている」奇妙な状態。
- 影響③: **バックテストと実機で挙動が変わりうる。** バー内で利益が一瞬TPに触れて戻る場合、サーバーTPは約定するがこの能動ロジックは無関係。Pythonマスター移植時、この死にコードを「動的利確」と誤解して再現すると実機とズレる。

### 2-2. 【FATAL-C】スプレッド制限フィルタが空回り（marketSpread未初期化）

ラボ規格でもコード自身でも「スプレッド制限」は生命線。だが**そのフィルタが機能していない**。

```cpp
// 529行: エントリー判定でスプレッドをチェック
if((marketSpread<=EA_SPREAD_LIMIT) && (lastAtr>=EA_ATR_VOLATILITY_LIM) ...
```

問題：`marketSpread`（TradeExpertクラスのメンバ変数、417行で宣言）は **`CheckTrade()` 内で一度も更新されていない**。

- `CheckTrade()` は `marketTickSize`, `marketAskPrice`, `marketBidPrice` は更新する（484-486行）が、`marketSpread` への代入が**どこにも無い**。
- メンバ変数の初期値は0（MQL4の数値メンバはデフォルト0）。
- → 条件 `marketSpread <= EA_SPREAD_LIMIT` は **`0 <= 3` で常に真**。

> **物理的帰結:** スプレッド制限フィルタは**常時パススルー**。スプレッドが何pips開いていようと、このEAは発注をブロックしない。「スプレッド制限を搭載」と見える防壁は、**コード上は宣言されているが実際には1度も作動しない死んだ防壁**。
> ラボ規格「高スプレッド回避必須」に対し、**回避機構が存在すると見せかけて存在しない**。前回検体（Generic）が「閾値が緩い防壁」だったのに対し、本検体は「**そもそも作動しない防壁**」。より悪質（無自覚なら、より危険）。

なお `TradeOrder` クラス側には別の `currentSpread`（133行）があり `updateMarketInfo()`（393-398行）で更新されるが、これは**エントリー判定の `marketSpread` とは別物**。判定に使われる変数だけが更新漏れしている。

### 2-3. 【FATAL-D】損益pips計算がShortで方向ミス（ASK/BID取り違え）

```cpp
// 181-193行 GetGrossProfitPips
double currPrice=MarketInfo(tradeSymbol,MODE_ASK);   // ← 常にASK
if(tradeOpenDirection==TRADE_LONG) {
   retVal=(int)((currPrice-tradeOpenPrice)/currentTickSize);
} else if(tradeOpenDirection==TRADE_SHORT) {
   retVal=(int)((tradeOpenPrice-currPrice)/currentTickSize);  // Short も ASK で評価
}
```

- Long は ASK で開いて（238行）、決済評価も ASK → スプレッド分だけ甘い評価だが方向は正しい。
- **Short は BID で開く（271行）のに、損益評価を ASK でやっている。** Short の含み損益評価には BID を使うのが正しい。ASK で評価すると、常にスプレッド分だけ不利（または有利）方向に系統誤差が乗る。
- §2-1で見た通り、この `GetGrossProfitPips` を使う能動決済はそもそも死んでいる（FATAL-A）ので実害は限定的だが、**コードの計算正しさ自体が破綻している**という物証。設計者が損益計算の方向性を正しく検証していない証左。

## 3. CAP（捏造）の徹底摘発

監査範囲：「ずる・偽り・嘘・未来予測・プラセボ・カーブフィッティング」を全行にわたって検証する。

### 3-1. 【FATAL-B】未来リーク（Look-ahead / Peeking）：**構造的に存在する**

これがCAP監査の最重要発見。**エントリー判定に未確定足（shift=0）を使っている。**

```cpp
// 488-497行
double stochMainCurr = iStochastic(..., 0, MODE_MAIN, 0);   // shift=0 ← 形成中の足
double stochSigCurr  = iStochastic(..., 0, MODE_SIGNAL, 0);  // shift=0
double maCurr     = iMA(tradeSymbol, EA_TIME_FRAME, 204, 0, EA_MA_METHOD, PRICE_CLOSE, 0);  // shift=0
double maCurrFast = iMA(tradeSymbol, PERIOD_M1, 4, 0, EA_MA_METHOD, PRICE_CLOSE, 0);         // shift=0
```

- `shift=0` は**現在形成中の、まだ確定していないバー**。このバーの値は次のティックで変わる（＝リペイント）。
- エントリー条件（535-573行）は `stochMainCurr`, `stochSigCurr`, `maCurr`, `fastMaSpeed`(=maCurrFast由来) を全て使う。**判定の中核が未確定値**。

**なぜこれがバックテストで致命的か（peekingの機序）:**

1. MT4ストラテジーテスターは、**every tick** モードでも各バーの中で価格を時系列に流すが、`shift=0` のインジ値は「そのティック時点での暫定値」を返す。
2. 問題は **control points / open prices モード**や、ティック生成が粗い場合。`shift=0` のStochクロスやMA傾きが「バー確定後の最終形」に近い値で評価されてしまい、**実機ではまだ確定していない情報を、テスターでは結果的に先読みした形**になりうる。
3. とくに `checkCrossing(stochPrev, stochCurr)` は「1本前(shift=1)と現在(shift=0)のクロス」を見るが、shift=0が動くと**同じバー内でクロス判定が出たり消えたりリペイントする**。テスターの時間解像度次第で「クロスが成立した瞬間の最良点」を拾える。

> **物理的帰結:** これは典型的な「**確定足を待たずに未確定足で判定する**」未来リーク。SEMURAラボの最重要鉄則「未来リーク禁止／確定bar[1]参照」に**真っ向から抵触**。
> バックテストが右肩上がりに見えていたなら、その相当部分は**この未来リークが生む幻覚**である可能性が高い。実機（リアルティック・確定遅延あり）では再現しない。
> **判定:** 悪意ある細工（時刻決め打ち等）とは色が違い、「初心者が `shift=0` と `shift=1` の意味を理解せず混在させた」実装ミス由来のリークと見るのが妥当。だが**結果としてバックテストを実態以上に良く見せる効果は同じ**。CAPの定義（事実の歪曲）に該当する。

補足の物証：同じコード内で `maPrev` は `shift=2`、`stochMainPrev` は `shift=1` を使っている（混在）。確定足の概念を一貫して適用できていない決定的証拠。

### 3-2. プラセボ（機能しない/無意味な指標・機能の羅列）：**複数該当**

コードに付随する説明文・コメントと、コードの死んでいる部分の対照：

### 3-3. カーブフィッティング（過剰最適化）の痕跡：**断定不可だが疑わしいマジックナンバー散在**

- Stochastic周期が **13,3,3**（標準は5,3,3 や 14,3,3）。13という半端な値は最適化由来の疑い。
- EMA **204**（200でも206でもない半端値）、EMA4(M1)、ATR周期4と14の混在。
- ATR閾値 `0.0003` / `0.0005`（通貨・桁依存の決め打ち。USDJPY等の円ペアでは桁が合わず意味をなさない可能性）。
- 取引時間 5-24時、最小間隔15分、タイムアウト30分。

### 3-4. Close-cap・損益操作：**該当なし（決済はサーバー約定ベース）**

## 5. ラボ規格への強制矯正による崩壊の数理的証明

**仮に上記バグを全て修正し、設計者の意図通り（ATR動的TP・固定SL）に動いたと仮定**し、ラボ規格に矯正した場合の崩壊を数式で示す。

### 5-1. デフォルト設定の実RR（矯正前の素の姿）

- TP = 2〜5pips（ATR分岐）、SL = 30pips。
- **リスクリワード = 利益2〜5 : 損失30 = 1:6 〜 1:15（利小損大）。**

損益分岐勝率 = `SL / (TP + SL)`：

| TP | SL | RR | 損益分岐勝率 |
|----|----|----|--------------|
| 5 | 30 | 1:6 | **85.7%** |
| 2 | 30 | 1:15 | **93.8%** |

> このEAは「**勝率85〜94%を出してトントン**」という構造。一見の高勝率は、損切りを6〜15倍遠くに置いて「滅多に負けないが、負けたら一撃で利益十数回分を失う」典型的な**利小損大の時限爆弾**。バックテストの高勝率はこの構造の必然であって、優秀さの証明ではない。

### 5-2. ラボ規格（TP5/SL5・RR1:1）への矯正後の崩壊

SLを30→5pipsに矯正し、RR1:1にする。スプレッド0.6pips（往復コスト相当）を適用。

**(a) 摩擦込みの損益分岐勝率:**
実効TP = `5 - 0.6 = 4.4`、実効SL = `5 + 0.6 = 5.6`（決済も摩擦を踏む悲観計算）。
損益分岐勝率 = `5.6 / (4.4 + 5.6)` = **56.0%**。

→ **ラボ目標勝率55%（=利益ゼロライン）を、スプレッドだけで既に超過。** 手数料・スリッページ前で赤字確定。

**(b) なぜ勝率55%維持が相場のフラクタル構造上不可能か（物理的根拠）:**

このEAの実勝率は、SLを30→5に縮めた瞬間に**激減する**。理由は、相場のノイズ構造にある：

1. **遅行指標の宿命的な出遅れ。** Stochastic(13,3,3) と EMA204/EMA4 はいずれも過去N本の平均・累積を使う遅行指標。シグナル確定は**値動きが起きた後**に出る。エントリーは必ず「動きの初動を逃した位置」になる。
2. **エントリー直後の自然なプルバック。** トレンド方向にエントリーしても、M5スケールでは**5pips程度の押し戻しは"ノイズ"の範疇**。GBPJPY/USDJPY等のM5平均的なバー振幅・ヒゲは数pips〜十数pipsあり、5pipsのSLは**1〜2本のノイズ足で容易に到達する**。
3. **フラクタル的自己相似性。** 価格変動は時間スケールを変えても統計的に自己相似（フラクタル）。M5の5pips幅のレンジ内往復は、より大きな足のノイズと同様、方向性を持たずランダムウォークに近い。**5pipsという至近のSLは、シグナルの優位性（エッジ）が顕在化する前に、ノイズだけで刈られる確率が高い。**
4. 数式で言えば：エッジが顕在化するのに必要な値幅を `D_edge`、SL距離を `D_sl` とすると、`D_sl << D_edge` のとき、**勝敗はエッジでなくノイズが決める**。本EAは遅行シグナルゆえ `D_edge` が大きく（初動を逃すため）、矯正後の `D_sl=5pips` は小さすぎる。→ 実勝率はランダムウォークの50%近傍へ収束し、56%の損益分岐を**構造的に超えられない**。

> **結論:** 遅行指標で初動を逃した位置から入り、5pipsという**ノイズ振幅と同オーダーのSL**を置くと、勝敗はエッジでなくノイズが支配する。実勝率は50%近傍に落ち、スプレッド込み損益分岐56%を超えられない。**勝率55%維持は、相場のフラクタル/ノイズ構造上、原理的に不可能。** これは設定ミスではなく、戦略思想（利小損大の遅行追随）とRR1:1規格の根本的非両立。

### 5-3. 極小TP×遅行指標×摩擦の期待値反転メカニズム

低ボラ時TP=2pipsのケースを物理計算する。

- スプレッド0.6pips。TP2pipsに対し、**入った瞬間に利幅の30%（0.6/2.0）が消える**。
- 実効TP = `2 - 0.6 = 1.4pips`。スリッページ（`EA_TRADE_MAX_SLIPPAGE=2`=2pips許容、薄い時間帯では現実的）を片側0.5pips乗せると実効TP=0.9pips。
- 一方、低ボラ判定でも SL は30pips（矯正前）or 5pips（矯正後）。
- **矯正後RR1:1（TP2/SL2に揃えた場合）:** 実効TP=1.4、実効SL=2.6 → 損益分岐勝率 = `2.6/(1.4+2.6)` = **65.0%**。スリッページ込みで70%超。
- 遅行指標で2pips抜きは、**シグナル確定の遅延だけで2pipsが消えるレベル**。Stochクロス→EMA加速確認→発注の多段ラグの間に、価格は既に2pips動いていることが常態。

> **結論:** TP2pipsは、スプレッド0.6で利幅の30%を即失い、スリッページと遅行ラグを加えると**実効利幅が1pips未満**まで縮む。損益分岐勝率は65〜70%超へ跳ね上がり、遅行指標の実勝率では到達不能。**極小TP×遅行指標×絶対摩擦は、期待値を構造的にマイナスへ反転させる。** 利益が出る理論的余地が、摩擦に物理的に食い尽くされる。
